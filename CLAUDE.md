# SmartPhoneSpecBattle

各種仕様やルールについては、コンテキスト情報の重複・肥大化を防ぐため「Less is More & Progressive Disclosure」の原則に基づき、以下の分割されたドキュメントを必要に応じて段階的に参照してください。

## 🤖 AI向け：最初に読むべきファイル (絶対ルール)
- 作業開始前に必ず **`docs/TODO.md`** を読み、現在の実装状況を把握すること。

## 📚 ドキュメント構成
- `README.md`: プロジェクトの目的と使用技術スタック等の概要
- `AGENTS.md`: Codex向けPRレビュー指示
- `docs/architecture.md`: アーキテクチャと設計方針（3層レイヤー構造）、状態管理とWidget分割の基準
- `docs/coding_rules.md`: 命名規則、Null Safety、コード構造の実装ルール
- `docs/TODO.md`: 現在の実装状況のサマリーと、次に取り組むべき残タスク
- `docs/SPECIFICATION.md`: 旧仕様書のアーカイブ（現行仕様の参照先は `docs/product_spec.md`）
- `docs/product_spec.md`: 現行プロダクトの仕様整理（要件定義の土台）
- `docs/gacha_browser_test_cases.md`: ガチャ機能のブラウザ結合テストケース
- `docs/device_info.md`: デバイス情報取得状況（プラットフォーム別取得可否・ステータス反映）
- `docs/plans/`: Planner/Generator/Evaluator ワークフロー用の計画・評価ドキュメント（運用ルールは `docs/plans/TEMPLATES.md` を参照）
- `PHASE4_SPEC_DRAFT.md`: 将来の機能拡張案（ガチャ・QR対戦等）の詳細仕様

**※注意:** 新規ドキュメントを作成する場合は `docs/` ディレクトリ内に配置し、このファイルから適宜リンクを追加してください。

## 🌲 Git Operation Rules
### 1. ブランチ戦略
- 全ての開発作業は `master` ブランチから分岐した `feature/` ブランチで行うこと。
- `master` ブランチへの直接のコミットは原則禁止とする。

### 2. プッシュ・マージポリシー
- `feature/` ブランチへのプッシュは許可する。
- 作業フロー: ブランチ作成 → 実装 → テスト/ビルド確認 → コミット → プッシュ → PR作成。
- PRは **Codex** にレビューを依頼し、指摘事項を修正した上でマージすること。
- プッシュ完了後、ブランチ名と変更内容のサマリーをユーザーに報告すること。

### 3. コミットメッセージ
- 日本語で記述し、変更内容がひと目でわかる簡潔な要約を含めること。

## 🧠 ワークフロー＆タスク管理の原則

### 1. ワークフロー設計と自己改善
- **計画ファースト**: 複雑なタスクは必ず詳細な仕様と計画から始め、行き詰まったら再計画する。
- **サブエージェント分割**: リサーチや並列タスクはサブエージェントへ委任し、メインコンテキストをクリーンに保つ。
- **自律的解決と自己改善**: バグは自らログを確認して完遂する。ユーザーからの修正・ルールは適切なドキュメントに記録し同じミスを防ぐ。
- **証明と品質担保**: 確実な動作証明（テスト・ログ確認）を完了条件とする。一時的なハックを避け、「エレガントかつシンプル」な根本的解決策を追求する。

### 2. タスク管理フロー
- **「計画 → 確認 → 実行記録 → ドキュメント化」** のサイクルを守ること。
- `docs/TODO.md` で計画と進捗を常に同期・更新し、高い透明性を提供する。

### 3. 検証手段（完了条件）
コードの追加や変更を行った後、タスクを「完了」とする前に、以下を実行しエラーゼロであることを確認すること。
- **静的解析**: `flutter analyze`
- **ユニットテスト**: `flutter test`

### 4. コア原則 (Core Principles)
- **シンプル第一**: 変更および影響するコードを最小限に抑える（過剰設計NG）。
- **手を抜かない**: 表面的な対処は避け、常にシニアエンジニア基準で根本原因を解決する。
