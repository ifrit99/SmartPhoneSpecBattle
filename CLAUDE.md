# SmartPhoneSpecBattle

各種仕様やルールについては、コンテキスト情報の重複・肥大化を防ぐため「Less is More & Progressive Disclosure」の原則に基づき、以下の分割されたドキュメントを必要に応じて段階的に参照してください。

## 🤖 AI向け：最初に読むべきファイル (絶対ルール)
- 作業開始前に必ず **`CONTEXT.md`** と **`docs/TODO.md`** を読み、AIエージェントの役割（Antigravity / Claude Code）や現在の実装状況を把握すること。
- **【重要】コンテナ開発の徹底**: 以降のFlutterに関するすべてのコマンド実行（`flutter analyze`, `flutter test`, `git add`, `git commit`, `git push` など）は、必ず提供されているDockerコンテナ内（`docker compose exec flutter-dev bash -c "..."`）で行うこと。ホストOS（Mac/VPSのローカル）での直接実行や権限要求は禁止する。

## 📚 ドキュメント構成
- `README.md`: プロジェクトの目的と使用技術スタック等の概要
- `docs/architecture.md`: アーキテクチャと設計方針（3層レイヤー構造）、状態管理とWidget分割の基準
- `docs/coding_rules.md`: 命名規則、Null Safety、コード構造の実装ルール
- `docs/TODO.md`: 現在の実装状況のサマリーと、次に取り組むべき残タスク
- `PHASE4_SPEC_DRAFT.md`: 将来の機能拡張案（ガチャ・QR対戦等）の詳細仕様

**※注意:** 新規ドキュメントを作成する場合は `docs/` ディレクトリ内に配置し、このファイルから適宜リンクを追加してください。

## 🌲 Git Operation Rules
### 1. 共通ブランチ戦略
- 全ての開発作業は `master` ブランチから分岐した `feature/` ブランチで行うこと。
- `master` ブランチへの直接のコミットは原則禁止とする。

### 2. 環境別プッシュポリシー
**A. ローカル環境（Mac / Antigravity）**
- **プッシュ禁止**: AIエージェントによるリモートリポジトリへの `git push` は一切禁止する。
- **作業範囲**: ブランチ作成、コード実装、ローカルでのコミットまでを担当すること。
- **最終確認**: プッシュおよびマージ操作は、ユーザーがGUIツールを用いて内容を検閲した上で手動で実行する。

**B. VPS環境（Claude Code）**
- **プッシュ許可**: 開発用ブランチ（`feature/`）に限り、リモートリポジトリへの `git push` を許可する。
- **作業フロー**: ブランチ作成 → 実装 → テスト/ビルド確認 → コミット → リモートへのプッシュ までを完結させること。
- **完了報告**: プッシュ完了後、ブランチ名と変更内容のサマリーをユーザーに報告すること。

### 3. コミットメッセージ
- 日本語で記述し、変更内容がひと目でわかる簡潔な要約を含めること。
