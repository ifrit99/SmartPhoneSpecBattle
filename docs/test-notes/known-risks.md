# 既知のリスクメモ（Known Risks）

断定ではなく「壊れやすい場所」「気をつけるべき場所」を列挙する。
改修や機能追加の前にざっと眺めるチェック材料として使う。

## 入力値・URLパラメータ系
- `?battle=` パラメータの値にスペース/`+`/欠損パディングが混入する可能性があり、`QrBattleService.extractBattleParam` / `normalizeBattleInput` 側で吸収している。UI側で別途 `trim()` せず、必ずドメイン層を経由する。
- v1（チェックサムなし）とv2（チェックサム付き）の両方を受け入れる設計。改ざん検知は v2 のみ機能するため、「v1も常に受け入れる」前提に依存した運用をしないこと。
- `generateShareUrl` は `Uri.base` を参照するため、呼び出し時点のブラウザURLに依存する。バックグラウンドで一括生成するケースは想定されていない。

## 表示と内部計算のズレ
- `BattleEngine` は事前計算、`BattleScreen` は再生型。計算結果と表示ログの整合が取れていないと「勝敗は出たがログが途中で止まる」等の不整合が起きうる。
- ダメージポップアップ・クリティカル演出は独立 StatefulWidget で動作するため、親の再構築と演出タイミングのズレに注意。
- 50ターン決着ロジックはHP割合→絶対値→敵勝利の優先度。表示上の「HPバー」と内部値にズレがあると誤解を生む。

## 境界値や未入力
- 初回起動時、`SharedPreferences` が空でも落ちないよう、`onboarding_completed` / `first_battle_completed` の未設定はfalse相当として扱う。
- デイリー報酬は日付跨ぎ復帰（アプリをバックグラウンド→日付変更→復帰）時にも再判定が必要。`WidgetsBindingObserver` が抜けると取りこぼす。
- ログイン報酬ポップアップはホーム画面が前面時のみ表示。他画面中は保留キューに積まれる。保留が消失しないよう、RouteAware の実装漏れに注意。
- ステータス生成で `DeviceSpecs` のいずれかが0/空のとき、`hashCode` の分布が偏り同じキャラが量産される可能性（仮置き）。
- バトル報酬はCPU対戦のみ付与。ゲスト対戦（QR/URL）でも誤って加算しないよう、経路分岐を壊さない。

## 仕様変更時に壊れやすそうな箇所
- `CharacterCodec` のバイナリフォーマットを拡張すると、既存URLが無効化される可能性。v3 以降を入れる場合は後方互換の戦略を先に決める。
- `ElementType` / `EffectType` の enum に値を追加した場合、既存のエンコード済みデータと `switch` 網羅のミスマッチが起きやすい。
- Web版サウンドは `document.createElement('audio')` 方式で安定化済み。`audioplayers` のメジャーバージョン更新時は再度Web挙動を検証する。
- `flutter` / `dart` のメジャーアップグレード時、Null安全の扱いや条件付きインポート構文（`platform_info_native.dart` / `platform_info_stub.dart`）が影響を受ける可能性。
- デバイス名は架空ブランドのみを表示（PR #13）。新規追加する敵やガチャキャラで実在メーカー名を使わない運用を徹底。
