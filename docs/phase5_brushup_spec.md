## 1. 要件定義

### 1-1. 背景と目的

Web版MVPはコアループ・継続導線・フレンド対戦まで完成しリリース可能な状態にある。
一方で次の3つのギャップが残っている。

1. **測れない**: 分析基盤・エラー監視がなく、継続率・共有率・ガチャ到達率などリリース後の判断材料を取得できない。
2. **サーバーが無い**: ランキング・クラウドセーブなど「次の価値」がすべてバックエンド不在でブロックされている。戦闘力表示（`PowerRatingService`）は `isEstimated` フラグでサーバー差し替えを前提に設計済み。
3. **磨き残し**: 空状態・エラー状態UXの総点検が未了、`home_screen.dart`（約3,500行）の肥大化が今後のUI変更速度を落とす。

Phase 5 のゴールは **「計測できる無料Web運用を確立し、サーバー導入の第一歩（ランキング）を踏み出す」**。

### 1-2. ゴール / 非ゴール

**ゴール**
- リリース後のKPI（初回バトル到達率、D1/D7継続率、URL共有率、ガチャ到達率）を計測可能にする
- 本番エラーを検知・追跡可能にする
- 全画面の空状態・エラー状態が「次に何をすべきか」を提示する
- 世界ランキング（週次リセット・任意参加）をサーバー実装で提供する
- `home_screen.dart` を保守可能な単位に分割する（挙動変更なし）
- バックアップコードにチェックサムを付与し、破損コード復元による データ破壊を防ぐ

**非ゴール（今回やらないこと）**
- 課金導線・決済・有料ジェム（Phase 4扱い。アカウント保存が前提）
- リワード広告（分析基盤で行動データを取得してから設計する）
- リアルタイム対戦・フレンドリスト
- Android/iOSネイティブビルド（ツールチェーン更新は別タスクのまま）
- 無料経済バランスの変更（`economy_balance_test.dart` の回帰基準は不変）

### 1-4. ユーザーストーリーと受け入れ基準

**F1: 分析基盤**
- 開発者として、初回起動〜初回バトル〜初回ガチャの到達ファネルを見たい。
  - 受け入れ基準: 本番環境で計測イベント（§2-1のイベント表）が分析コンソールに到達する。
- プレイヤーとして、計測されることを起動時に知り、拒否できる。
  - 受け入れ基準: 同意前にイベントが1件も送信されない（`first_visit` / `session_start` 等のSDK自動収集イベントを含む）。拒否してもゲームは全機能遊べる。

**F2: エラー監視**
- 開発者として、本番の未捕捉例外をスタックトレース付きで受け取りたい。
  - 受け入れ基準: 意図的なテスト例外がリリースビルドのWebから収集サービスに届く。個人情報（バックアップコード・共有URL全文）がペイロードに含まれない。

**F3: 空状態・エラー状態UX**
- 新規プレイヤーとして、データが無い画面でも「次に何をすればよいか」が分かる。
  - 受け入れ基準: §2-3の対象一覧の全項目が「状態の説明＋次アクション導線」を持つ。

**F4: ホーム画面分割**
- 開発者として、ホームのカード1枚の修正が1ファイルの変更で済むようにしたい。
  - 受け入れ基準: `home_screen.dart` が800行以下になり、各カードが `lib/presentation/widgets/home/` 配下の独立Widgetになる。画面の見た目・挙動は変わらない（スクリーンショット比較）。

**F5: バックアップv2**
- プレイヤーとして、コピーミスしたバックアップコードで復元してもデータが壊れない。
  - 受け入れ基準: 1文字破損したv2コードは復元前に「コードが破損しています」と拒否される。既存v1コードは引き続き復元できる。

**F6: オンラインランキング MVP**
- プレイヤーとして、自分のスマホ（戦闘力）が世界で何位・上位何%かを実数で知りたい。
  - 受け入れ基準: ランキング参加をONにすると自分のスコアが送信され、上位一覧と自分の順位・上位%が表示される。ホームの `PowerRatingCard` の「推定」表記が実測に置き換わる（`isEstimated: false`）。
- カジュアル方針の反映: 参加は任意（デフォルトOFF）。週次リセット。順位報酬なし（表示と称号のみ）。未参加・オフライン時は従来の推定表示へ自動フォールバック。

### 1-5. 非機能要件

| 項目 | 要件 |
|------|------|
| プライバシー | 計測・ランキングとも任意参加。PII（氏名・メール等）は一切収集しない。ユーザー識別子は **Firebase Anonymous Auth が発行する匿名UID** のみ（ランキング参加時に初めて取得。Firestoreセキュリティルールの `request.auth.uid` 照合に使用し、PIIとの紐付けは行わない） |
| オフライン耐性 | サーバー機能（ランキング）が到達不能でも既存機能は一切劣化しない（推定表示へフォールバック） |
| パフォーマンス | 追加SDK（分析・監視）による初期ロード増加を計測し、`flutter build web` の main.dart.js サイズ増を記録する（目安: +15%以内） |
| コスト | サーバーは無料枠内で運用できる構成を選ぶ（詳細は §2-4） |
| 後方互換 | バックアップv1コード・キャラクターコーデックv1/v2/v3の復元互換を維持 |

## 2. 機能設計

3層構造（`lib/data` / `lib/domain` / `lib/presentation`）と ServiceLocator パターンは現行踏襲。状態管理は引き続き `StatefulWidget + setState`。

### 2-1. F1: 分析基盤

**技術選定（推奨: Firebase Analytics）**
- 理由: Flutter Web対応の公式プラグイン（`firebase_analytics`）があり、F6/F7で使うFirebase（Firestore・匿名認証）と基盤を共有できる。GA4コンソールで閲覧可能。
- 代替案: gtag.js 直接埋め込み（依存は軽いがF6と基盤が分かれる）。

**構成**
- `lib/domain/services/analytics_service.dart`: 抽象インターフェース。`logEvent(name, params)` と同意状態管理（`isEnabled`）。ドメイン層はFirebaseに依存しない。
- `lib/data/firebase_analytics_client.dart`: 実装。Firebase Analyticsは**収集無効の状態で初期化する**（`setAnalyticsCollectionEnabled(false)` ＋ Consent Mode を既定 `denied` で設定）。同意取得後に初めて収集を有効化することで、`first_visit` / `session_start` 等のSDK自動収集イベントも同意前には送信されない。未同意・拒否時はラッパー側でも全イベントを破棄する（二重ガード）。
- `ServiceLocator` に登録。テストでは No-op 実装に差し替え。
- 同意フラグは SharedPreferences（`analytics_consent`: 未回答/許可/拒否 の3値）。バックアップ対象外（端末ごとの判断）。

**計測イベント一覧（初期セット）**

### 2-2. F2: エラー監視

- `sentry_flutter`（Web対応）を採用。`main()` を `SentryFlutter.init` でラップし、`FlutterError.onError` と非同期例外を捕捉。
- release タグに git SHA（CIの `GITHUB_SHA`）を注入し、デプロイと紐付ける。
- `beforeSend` でバックアップコード・`?battle=` パラメータ値をマスキング（URLはパス部のみ送信）。
- 分析同意とは独立に、エラー送信は技術的必須として同意ダイアログ内で明示する（文言は §3-2）。
- サンプルレート: エラー100% / パフォーマンストレースは無効（コスト・サイズ節約）。
- **リリース順序の制約**: Sentryの有効化は、F1の同意ダイアログ（エラー送信の明示）が本番デプロイされた後に行う。コードを先にマージする場合はDSN未設定または無効化フラグで送信を止めた状態でリリースし、通知UIなしでエラー送信が始まる状態を作らない。

### 2-4. F6: オンラインランキング MVP

**バックエンド選定（推奨: Firebase / Firestore + 匿名認証）**
- 理由: 無料枠が十分（読取5万/日）、F1のFirebase導入と基盤共有、匿名認証がF7（クラウドセーブ）へそのまま拡張できる。サーバーコードのデプロイ・運用が不要（セキュリティルールのみ）。
- 代替案: Cloudflare Workers + KV（軽量だが認証・集計を自作）、Supabase（RLSは強力だがFirebaseと二重基盤になる）。

**データモデル（Firestore）**

```
rankings/{weekId}/entries/{anonUid}:
  powerRating: int        // PWRスコア
  characterCode: string   // CharacterCodec v3（サーバー側検証・アバター表示用）
  title: string           // プレイヤー称号
  updatedAt: timestamp
  expiresAt: timestamp    // TTLポリシー対象。updatedAt + 30日（失効時刻。updatedAt自体をTTL対象にすると即時失効するため別フィールド）
  uid: string             // anonUidの複製（参加解除時に全週entryをcollectionGroupクエリで削除するため）
```

- `anonUid` は **Firebase Anonymous Auth のUID**（ランキング参加ON時にサインインして取得。クライアント生成のUUIDは使わない）。
- `weekId` はローカルリーグと同一の週ID計算を使用（週次リセット）。
- 書き込みはセキュリティルールで制約: `request.auth.uid == anonUid` を強制（他人のエントリへの書き込み拒否）/ powerRating は 0〜理論上限（全端末Lv上限×覚醒+5から算出した定数）/ characterCode 長さ上限。
- 順位取得（3クエリ）: ①上位50件の取得（一覧表示用）、②自分より高スコアの件数の `count()` 集計 → 順位 = 件数+1、③`rankings/{weekId}/entries` 全体の `count()` 集計 → 総参加数N。上位% = 順位 ÷ N × 100 とし、「上位○%（参加N人中）」（§3-3）の分母はNを使う（上位50件を分母にしない）。
- 参加解除（OFF）: **過去週分を含む全週の自分のentryを即時削除する**（「OFF=サーバーからデータが消える」というプライバシー期待に応える）。`collectionGroup('entries').where('uid', '==', 自分のuid)` で全週分を取得して一括削除し、当週分の削除により一覧・順位・総参加数からも即時除外される。削除権限はセキュリティルールで `request.auth.uid == resource.data.uid` のみに制限（collectionGroup経由の読取も同条件）。
- `expiresAt`（`updatedAt` + 30日）のFirestore TTLは、解除操作を経ずに離脱したユーザーの過去週entryを自動削除する**保険**として設定する（解除時削除の代替ではない）。

**不正対策の割り切り（既知リスク）**
- PWRはクライアント計算のため完全な防止は不可能。MVPでは (a) ルールでの値域クランプ、(b) `characterCode` のチェックサム検証を将来のCloud Functionsで実施できる形でデータを残す、(c) 順位に報酬を付けない（カジュアル方針と整合）ことで実害を抑える。

**ドメイン層**
- `lib/domain/services/ranking_service.dart`: 参加登録/解除、スコア送信、上位一覧＋自分の順位取得。結果は `RankingSnapshot`（entries, myRank, myPercentile, isEstimated=false）。
- 失敗時（オフライン/未参加）は `PowerRatingService` の既存推定値（`isEstimated=true`）を返すフォールバックを `RankingService` 内で完結させる（UIはデータ取得経路＝サーバー/ローカルを意識しない）。
- **表示切替（UI側の必須変更）**: 現行 `lib/presentation/widgets/power_rating_card.dart` は「推定上位○%」表記とローカル比較の免責文をハードコードしており `isEstimated` を参照していない。F6では `isEstimated` に応じたラベル切替を実装する — `true`: 現行どおり「推定上位○%」＋免責文 / `false`: 「世界○位（上位○%）」＋週次リセット表記（免責文は非表示）。ランキング詳細シート側も同じ規則に従う。
- 送信タイミング: ホーム画面表示時に、前回送信ペイロード（`lastSentWeekId` / `powerRating` / `characterCode` / `title` をローカル保存）のいずれかが変化していれば送信（デバウンス、1セッション最大数回）。**週替わり（weekIdの変化）時はPWRが同一でも必ず再送**し、新しい `rankings/{weekId}` へ登録する（参加ONなのに今週のランキングから消える事態を防ぐ）。

### 2-5. F4: ホーム画面分割（挙動変更なし）

- 分割先: `lib/presentation/widgets/home/`。カード単位で1ファイル（例: `daily_mission_card.dart`, `season_pass_card.dart`, `rival_road_card.dart`, `power_rating_card.dart`, `next_action_card.dart` 等、現行ホームの機能カードすべて）。
- 設計原則: 各カードは **Stateless**。表示データとコールバックをコンストラクタで受け取り、状態・永続化は従来どおり `HomeScreen` が保持する（状態管理方式を変えない）。
- 進め方: 1PRあたり2〜4カードの段階分割。各PRで `flutter analyze` / `flutter test` / スクリーンショット比較（変化なしを確認）。
- 完了定義: `home_screen.dart` ≦ 800行。

### 2-6. F5: バックアップコード v2

- 現行: `SPEC-BATTLE-BACKUP:` + Base64url JSON（整合性検証なし）。
- v2 エンコード手順（`CharacterCodec` v2 と同一方式・実装を再利用）:
  1. JSONペイロードをUTF-8バイト列化する
  2. そのバイト列のHMAC-SHA256を計算し、**先頭4バイト**をチェックサムとする
  3. `ペイロードバイト列 + チェックサム4バイト` を連結し、**まとめて1回だけBase64url化**する
  4. 先頭に `SPEC-BATTLE-BACKUP2:` プレフィックスを付与する
- v2 復元手順: プレフィックス除去 → Base64urlデコード → 末尾4バイトをチェックサムとして分離 → 残りのバイト列からHMAC-SHA256先頭4バイトを再計算して比較。不一致なら `IntegrityException`（F3-7の文言で拒否）。
- v1プレフィックスは従来どおり復元可（後方互換）。生成は常にv2。
- 既知の限界: HMAC鍵はクライアント埋め込みのため改ざん防止ではなく**破損検知**が目的（`product_spec.md` 7-1と同じ制約）。

### 2-7. F7/F8（P2・設計メモのみ、今回実装しない）

- **F7 クラウドセーブ**: F6の匿名認証UIDをキーに、バックアップコードと同一のJSONペイロードをFirestoreへ保存/復元。手動バックアップの上位互換として提供し、既存のコード方式も残す。
- **F8 動的OGP**: `?battle=` URLに対し、エッジ関数（Cloudflare Workers等）でキャラ名・PWR入りOGP画像を生成。SNS共有のクリック率向上が狙い。F1で `share_url_opened / share_url_created` 比を測ってから投資判断する。

## 4. テスト観点

完了条件は全機能共通で `flutter analyze` エラー0 / `flutter test` 全パス（CLAUDE.md準拠）。以下は機能固有の観点。

### 4-1. ユニットテスト

| 対象 | 観点 |
|------|------|
| AnalyticsService | 未同意時にイベントが送信されない / 同意後に送信される / 3値の同意状態遷移 |
| RankingService | 送信デバウンス / **週替わり時はPWR不変でも再送される** / オフライン時に推定値フォールバック / 順位・上位%計算（分母が総参加数Nであること。参加51人以上のケースを含む） / 参加解除時に過去週を含む全週の自分entryが削除され推定表示へ戻る / weekId計算がローカルリーグと一致 |
| バックアップv2 | 正常往復 / 1文字破損で `IntegrityException` / v1コード復元互換 / v2生成プレフィックス |
| 経済バランス | **既存 `economy_balance_test.dart` が無変更でパス**（カジュアル方針: 経済に手を入れていない証明） |
| コーデック互換 | Character v1/v2/v3 デコード互換の既存テスト維持 |

### 4-2. Widgetテスト

| 対象 | 観点 |
|------|------|
| 分割後ホームカード | 各カードが入力データを正しく描画 / コールバック発火（分割PRごとに追加） |
| EmptyStateCard | 文言・導線ボタンのタップで期待画面へ遷移 |
| 同意ダイアログ | 初回のみ表示 / 選択の永続化 / 「協力しない」後もホームへ遷移 / `?battle=` 直リンク起動でも未回答ならゲストプレビュー前に表示 |
| ランキングシート | 参加ON/OFF切替 / 自分の行のハイライト / 未参加時の推定表示 |
| PowerRatingCard | `isEstimated=true` で「推定上位○%」＋免責文 / `false` で「世界○位（上位○%）」＋週次リセット表記に切り替わる |

### 4-3. ブラウザ結合テスト（手動・Playwright）

- 初回起動 → 同意ダイアログ → 拒否 → 全コアフロー（バトル/ガチャ/共有）が動作すること
- 初回ユーザーが `?battle=` 付きURLで直接アクセスした場合も、ゲストプレビュー前に同意ダイアログが表示され、回答前に `share_url_opened` が送信されないこと
- 未同意・拒否状態でDebugViewに**自動収集イベント（`first_visit` / `session_start` 等）を含めて**1件も到達しないこと
- 同意 → 分析コンソール（DebugView）で `battle_start` 〜 `battle_result` の到達確認
- ランキング: 2ブラウザ（別プロファイル）で参加し、相互の順位が反映されること。一方が参加OFFにすると、もう一方の一覧・総参加数から即時消えること
- オフライン（DevToolsでOffline化）: ホーム表示・バトル・ガチャが劣化なく動き、ランキングが推定表示に戻ること
- ホーム分割PRごと: 分割前後のスクリーンショット比較で差分ゼロ
- バックアップ: 旧バージョンで作成したv1コードの復元 / v2コード破損時の拒否
- 意図的例外（デバッグ用隠し操作 or 一時コード）がSentryに届き、`?battle=` 値がマスクされていること

### 4-4. 非機能・回帰

- `flutter build web` 成功、main.dart.js サイズを F1/F2導入前後で記録（+15%以内）
- 初期ロード時間の目視比較（GitHub Pages本番）
- 既存テスト全件（370件超）パス
- Firestoreセキュリティルールのテスト（他人のuidへの書き込み拒否 / **他人のentryの削除拒否・自分のentryの削除許可** / 値域外powerRating拒否 / `expiresAt` 欠落エントリの書き込み拒否）— Firebase Emulator Suite
