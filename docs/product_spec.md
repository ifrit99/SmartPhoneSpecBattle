## 1. プロダクト概要

### このアプリは何をするものか

「SPEC BATTLE」は、**スマートフォンの実スペック（CPU・RAM・ストレージ・バッテリー残量）をキャラクターのステータスに変換し、ターン制オートバトルで戦わせる**ブラウザゲームである。

プレイヤーは自分の端末スペックから自動生成されたキャラクターを使い、CPU敵と戦ったり、URLを共有してフレンドと対戦したりできる。加えて、ガチャシステムにより架空のスマートフォンスペックからキャラクターを獲得・育成する要素もある。

### 想定ユーザー

- スマートフォンに興味のあるカジュアルゲーマー
- 自分の端末スペックを「遊び」に変えたいユーザー
- 友人とURLを共有して気軽に対戦したいユーザー
- **要確認**: ターゲット年齢層・地域は未定義

### 現在提供している主な価値

| 価値 | 説明 |
|------|------|
| 新規性 | 「端末スペック＝キャラ性能」という独自のコンセプト |
| 手軽さ | ブラウザだけで遊べる（インストール不要、PWA対応） |
| ソーシャル性 | URL共有でフレンド対戦が可能 |
| コレクション性 | ガチャで架空端末キャラを収集・図鑑で確認 |

## 2. 主要機能一覧

### 2-1. キャラクター自動生成

| 項目 | 内容 |
|------|------|
| 概要 | 端末のハードウェア情報からキャラクターを決定論的に生成する |
| ユーザーができること | アプリ起動時に自分の端末キャラが自動で作られる |
| 生成ルール | CPU→ATK、RAM→HP、ストレージ空き→DEF、バッテリー残量→SPD |
| 属性 | OS バージョン文字列から6属性（炎・水・地・風・光・闇）の1つを決定 |
| スキル | 属性ごとに固定3つ（攻撃系・防御系・特殊系） |
| 外見 | 頭・体・腕・脚・カラーの組み合わせによるピクセルアート（自動生成） |
| 名前 | 属性ベースの接頭語＋ランダム接尾語（例: フレア・ナイト） |
| 補足 | 同一スペックからは常に同一キャラが生成される（シード値による決定性） |
| 補足（Web） | Web版ではデバイス情報APIの制約あり。取得可能な範囲でフォールバック動作 |

### 2-2. CPU対戦（ソロバトル）

| 項目 | 内容 |
|------|------|
| 概要 | ランダムに生成されたCPU敵とオートバトルで戦う |
| ユーザーができること | ホーム画面の「バトル開始」から対戦。敵プレビューで解析と推奨戦術を確認し、バトル画面で支援コマンドを選んで開始 |
| 敵の種類 | 16種の架空端末（4難易度×4体）から確率抽選 |
| 難易度分布 | Easy 20%、Normal 50%、Hard 25%、Boss 5% |
| 高難度挑戦 | ホームのチャレンジ導線からHARD/BOSSを明示選択して挑戦可能。BOSS勝利は1日1回だけ専用撃破報酬を追加付与 |
| BOSSタイムアタック | CPU BOSS勝利時の最短ターンを自己ベストとして保存し、記録更新時はリザルトに表示 |
| ライバルロード | 固定ライバル5戦を順番に攻略する常設PvE進行。各ステージ初回突破時にCoin/Gems報酬を自動付与し、ステージ別最短ターンを保存 |
| 支援コマンド | バトル開始時に「支援なし」「攻撃支援」「防御支援」を選択。攻撃支援は3ターン攻撃/速度強化、防御支援は3ターン防御/再生を付与 |
| 報酬 | 経験値（勝敗問わず）＋コイン（勝利時に難易度ボーナス加算） |
| バトル方式 | 完全オート、最大50ターン、SPD順で行動、AI行動選択 |
| 補足 | プレイヤーの操作介入はスキップのみ |

### 2-3. フレンド対戦（URL共有）

| 項目 | 内容 |
|------|------|
| 概要 | 自分のキャラデータをURLにエンコードし、相手に共有して非同期対戦する |
| ユーザーができること | ①URLを発行してコピー/共有 ②受け取ったURLを貼り付けて対戦 |
| データ形式 | キャラクターのバイナリデータ→Base64url→URLクエリパラメータ |
| 改ざん防止 | HMAC-SHA256ベースの4バイトチェックサム（v2形式） |
| 後方互換 | v1形式（チェックサムなし）のデータもデコード可能 |
| URL形式 | `https://{host}/?battle={encoded_data}` |
| 補足 | サーバーレス設計。データはURL自体に含まれ、バックエンドは不要 |

### 2-4. ガチャ（エミュレートガチャ）

| 項目 | 内容 |
|------|------|
| 概要 | ゲーム内通貨（コイン）を消費し、架空端末スペックからキャラクターを獲得する |
| ユーザーができること | 1回引き（100コイン）または10連（900コイン、SR以上1体確定）。結果画面から獲得キャラを直接装備できる |
| プレミアム解析 | 20ジェムで1回引き。SR以上確定（SR 67% / SSR 33%）。SSR抽選時は日替わりピックアップSSRが60%で出現し、5回連続でピックアップSSRを外すと次回はピックアップSSR確定 |
| イベント解析 | 30ジェムで1回引き。SR以上確定で、週替わり限定SSRが25%で出現。3回連続で限定SSRを外すと次回は限定SSR確定 |
| 解析ロードマップ | ホーム画面で限定SSR/日替わりSSRの対象端末、天井までに必要なGems、不足Gems、優先ターゲットを表示 |
| レアリティ | N(60%)、R(25%)、SR(10%)、SSR(5%) |
| ステータス倍率 | N:0.8倍、R:1.0倍、SR:1.2倍、SSR:1.5倍 |
| ガチャプール | 17種の架空端末（N:5、R:5、SR:4、SSR:3） |
| 10連天井 | 最初の9体にSR以上がなければ10体目はSR(67%)/SSR(33%)確定 |
| 重複覚醒 | 同一レアリティ・同一端末を重複入手した場合は新規枠ではなく既存キャラの覚醒に変換。最大+5まで基礎ステータスが上昇 |
| 上限後補填 | 覚醒+5到達後の同一キャラ重複はレアリティ別コイン補填に変換（N:25、R:45、SR:100、SSR:180） |
| ロスター収集ボーナス | 端末種、SSR、限定端末、覚醒合計に応じてCPU戦のCoin報酬倍率が最大+20%まで上昇。編成画面で進捗を表示 |
| 補足 | 課金導線は未実装。ジェムはデイリー報酬で獲得し、プレミアム解析・イベント解析に使用する |

### 2-5. 編成（インベントリ）

| 項目 | 内容 |
|------|------|
| 概要 | ガチャで入手したキャラから1体を「装備」してバトルに使用できる |
| ユーザーができること | キャラ一覧の閲覧、詳細確認、装備/解除 |
| ソート/絞り込み | レアリティ降順 → レベル降順。レアリティ別フィルタあり |
| 分析表示 | 所持数、端末種類数、SSR数、最高Power、装備中キャラを表示 |
| 空状態 | 未所持時は所持通貨とガチャ導線を表示し、そのまま獲得画面へ移動できる |
| 補足 | 装備するとホーム画面・バトル・URL共有すべてで装備キャラが使われる |

### 2-6. コレクション（図鑑・戦績）

| 項目 | 内容 |
|------|------|
| 概要 | 過去に遭遇した敵キャラの図鑑、プレイヤーの対戦成績、ローカルランク、ローカルリーグ、直近バトル履歴、実績報酬を閲覧できる |
| ユーザーができること | 敵キャラ図鑑（発見/未発見表示）、戦績（勝率・勝利数等）、ランク進捗、固定ライバルとのローカル順位、直近20件のバトル履歴、実績進捗、達成報酬の受取 |
| ローカルランク | 勝利数、バトル数、敵発見数、ガチャロスター、限定端末所持、ライバルロード進捗、BOSS最短記録からランクポイントを算出。スペックハンター以上の各ランク到達時にCoin/Gems報酬を一度だけ受け取れる |
| ローカルリーグ | 週IDごとの固定ライバル順位表にプレイヤーのRPを混ぜ、次に抜く相手と必要RPを表示 |
| バトル履歴 | 勝敗、相手、CPU/フレンド区分、難易度、ターン数、戦術、支援コマンド、獲得報酬を保存 |
| 実績 | 初バトル、初勝利、バトル回数、勝利数、敵発見数、ガチャ所持数、ライバルロード突破などの短期/中期目標 |
| 補足 | 未発見キャラは「???」表示で、発見意欲を刺激する設計 |

### 2-7. レベル・経験値システム

| 項目 | 内容 |
|------|------|
| 概要 | バトル結果に応じて経験値を獲得し、レベルアップでステータスが上昇する |
| 経験値の付与先 | 実機キャラ使用時はプレイヤー共通レベル、ガチャキャラ装備時はその装備キャラのみ成長 |
| レベルアップ | 必要経験値 = `100 × (1.0 + (Lv-1) × 0.5)` で漸増 |
| ステータス上昇 | 全ステータスに `1.0 + (Lv-1) × 0.1` の倍率（Lv10で約1.9倍） |
| 経験値獲得 | 勝利: `50 × (1.0 + (敵Lv-1) × 0.2)`、敗北: `20 × 同倍率` |

### 2-8. 通貨システム

| 項目 | 内容 |
|------|------|
| 概要 | バトル報酬としてコインを獲得し、ガチャに使用する |
| コイン獲得量 | 基本: `30 + Lv × 5`、勝利時に難易度ボーナス（+0〜+50） |
| プレミアムジェム | デイリーログイン・CPUバトル報酬で獲得。20ジェムでプレミアム解析、30ジェムでイベント解析に使用 |
| 連続ログイン | 7日サイクルのストリークを記録。3日目は+10ジェム、7日目は+20ジェムの追加ボーナス |
| デイリーミッション | バトル1回、勝利1回、ガチャ1回の3条件を日次で記録。達成後にコイン/ジェム報酬を個別または一括で受け取れる |
| 日替わりショップ | Coinを消費して、装備キャラEXP、Battery回復、少量Gems交換を1日1回ずつ購入できる |
| 週次チャレンジ | CPUのHARD/BOSS勝利を週3回達成すると+600コイン/+50ジェムを受け取れる |
| 期間イベント | 週替わりイベントを開催。ホームから今週の専用ライバルに挑戦でき、専用ライバル勝利は2勝分進む。2勝/4勝の段階報酬と、CPU戦5勝のイベント別コイン/ジェム報酬を受け取れる |
| シーズンパス | バトルごとにシーズンポイントを獲得。月替わりの報酬トラックでCoin/Gemsを段階受取できる |
| ライバルロード | 5ステージの常設ライバルを順番に倒すと一度きりのコイン/ジェム報酬を受け取れる。再戦ではステージ別最短ターン更新を狙える |
| 初回撃破ボーナス | 未撃破のCPU敵に初勝利すると+120コイン/+10ジェムを自動付与 |
| BOSS撃破報酬 | CPU BOSSに勝利すると1日1回だけ+300コイン/+30ジェムを自動付与 |

### 2-8-1. 経済バランスの回帰基準

| 基準 | 内容 |
|------|------|
| 初回セッション | 通常CPU勝利1回＋日次バトルミッションで単発ガチャ1回分、ログイン＋CPUバトル報酬＋勝利ミッションでイベント解析1回分に届く |
| 通常周回 | Lv1通常CPU勝利3回で単発ガチャ1回分のコインに届く |
| 7日間の継続 | ログイン・CPUバトル・勝利ミッションを7日分こなすと、プレミアム解析天井1回分とイベント解析天井1回分の合計ジェムに届く |
| 週替わりイベント | CPU5勝のイベント報酬は、最低報酬週でもイベント解析2回分以上のジェムを返す。専用ライバル勝利は2勝分として進み、2勝/4勝時点でも途中報酬を受け取れる |
| 週次チャレンジ | HARD/BOSSの週3勝報酬は単発ガチャ6回分以上のコインとプレミアム解析2回分以上のジェムを返す |
| シーズンパス | 月間15回の通常CPU勝利で完走圏内。完走報酬は10連2回分以上のコインとイベント解析天井1回分以上のジェムを返す |

### 2-9. サウンド

| 項目 | 内容 |
|------|------|
| 概要 | BGM（バトル中ループ再生）とSE（攻撃・防御・スキル・勝敗等）を再生 |
| ミュート | BGMとSEを個別にON/OFF可能。設定はSharedPreferencesに保存され、再起動・バックアップ復元後も反映される |
| Web対応 | AutoPlayポリシー対応済み（初回タップでAudioContext解放）。音声アンロック失敗時も画面進行はブロックしない |
| SE再生方式 | Web版はHTML5 Audio APIを直接使用（audioplayers_webのリソース枯渇問題を回避） |

## 4. ユーザーフロー

### 4-1. 初回起動〜初バトル（正常系）

```
1. ブラウザでアクセス
2. タイトル画面表示
   - Web: 1回目のタップでAudio解放 + BGM開始
   - Web以外/2回目以降のタップで次画面へ遷移
3. ホーム画面表示
   - 端末スペックを読み取り、キャラクターを自動生成
   - キャラカードに名前・属性・レベル・ステータスが表示される
4. 「バトル開始」ボタンをタップ
5. 敵プレビューシート表示
   - 敵キャラの名前・難易度・ステータス・端末情報を確認
6. 「バトル！」をタップ
7. バトル画面に遷移
   - 支援コマンドを選択してから、オートでターン進行、ダメージポップアップ・SE演出
   - スキップ可能
8. バトル終了 → 「リザルトへ」をタップ
9. リザルト画面
   - 勝敗、獲得EXP・コイン表示
   - レベルアップ時はアニメーション演出
10. 「ホームへ戻る」→ ホーム画面に帰還（データ更新済み）
    - CPU対戦では「もう一戦」で次の敵プレビューへ直行できる
    - ガチャ可能な通貨がある場合は「ガチャ」でガチャ画面へ直行できる

高難度挑戦:
- ホームの「高難度チャレンジ」からHARD/BOSSを選択
- 選択した難易度の敵プレビューを表示
- 勝利時は難易度ボーナス込みのコイン報酬を獲得
```

### 4-2. ガチャ〜キャラ装備

```
1. ホーム画面 → 「Gacha」ボタン
2. ガチャ画面でレート確認
3. 「1回引く」（100コイン）or「10連」（900コイン）をタップ
4. ガチャマシンのシェイク演出
5. 結果ダイアログ表示（レアリティ・キャラ名・ステータス。重複時は覚醒または上限後補填表示）
6. 結果画面から直接装備、または「OK」で閉じる
7. 「Party」ボタン → 編成画面
8. 獲得キャラをタップ → 詳細シート → 「このキャラクターで戦う」
9. ホーム画面に戻ると装備キャラが反映されている
```

### 4-3. フレンド対戦（URL共有側）

```
1. ホーム画面 → 「Friend」ボタン
2. フレンドメニュー → 「URLでシェアする」
3. シェア画面にURL表示
4. 「コピー」or「共有」ボタンでURLを取得
5. SNS/メッセンジャーで友人に送信
```

### 4-4. フレンド対戦（URL受取側）

```
パターンA: URL直接アクセス
1. 受け取ったURL（?battle=xxx付き）をブラウザで開く
2. アプリ起動 → URL解析 → ゲストプレビュー画面に直接遷移
3. 自分vs相手の相性・推奨戦術を確認
4. 戦術をそのまま使う、または変更 → 「バトル開始」→ バトル → リザルト

パターンB: 手動入力
1. アプリ内でフレンドメニュー → 「URLを入力して対戦」
2. URLまたはコードを貼り付け → 読み取り状態を確認 → 送信
3. ゲストプレビューで相性分析と戦術を確認 → バトル → リザルト
```

### 4-5. 異常系・空状態

| 状況 | 挙動 |
|------|------|
| コイン/ジェム不足でガチャ | スナックバーで不足量と入手導線を表示、ボタンは無効化されない |
| 不正なURL/コード入力 | 赤色エラーテキストをインライン表示 |
| 改ざんされたURLデータ | `IntegrityException` → エラー表示（チェックサム検証失敗） |
| ガチャキャラ0体で編成画面 | 専用の空状態メッセージを中央表示 |
| 図鑑で全キャラ未発見 | 全カード「???」表示 |
| バトルが50ターンで決着しない | 残HP割合が高い方を勝者とする。同率なら敵勝利 |

## 5. データ構造・状態管理の概要

### 5-1. 主要データモデル

| モデル | 主要フィールド | 役割 |
|--------|--------------|------|
| Character | name, element, baseStats, currentStats, skills, experience, seed, batteryLevel, 外見インデックス×5 | ゲームの中核。バトル・表示・エンコードすべてに使用 |
| Stats | hp, maxHp, atk, def, spd | 戦闘ステータス（不変値オブジェクト） |
| Skill | name, element, category, multiplier, cooldown, effect | キャラの行動選択肢（属性ごとに3つ固定） |
| StatusEffect | type, duration, value, isPermanent | バフ・デバフ（9種: ATK/DEF/SPD増減、毒、再生、スタン） |
| Experience | level, currentExp, expToNext | レベル進行管理 |
| GachaCharacter | id, deviceName, rarity, obtainedAt, awakeningLevel, character | ガチャ専用ラッパー（レアリティ・覚醒情報付き） |
| PlayerCurrency | coins, premiumGems | 通貨管理 |
| DecodedCharacter | character, isGacha, rarity, deviceName | URLデコード結果のラッパー |

### 5-2. データの流れ

```
[端末スペック] → CharacterGenerator → Character
[ガチャプール] → GachaService → GachaCharacter → Character
[Character]   → BattleEngine  → BattleResult（ログ・EXP・コイン）
[Character]   → CharacterCodec → Base64url文字列 → URL
[URL]         → CharacterCodec → DecodedCharacter → Character
```

### 5-3. 永続化

| データ | 保存先 | 方式 |
|--------|--------|------|
| プレイヤーレベル・経験値 | SharedPreferences | Key-Value |
| 戦績（勝利数・バトル数） | SharedPreferences | Key-Value |
| 直近バトル履歴 | SharedPreferences | JSON文字列リスト（最大20件） |
| 通貨（コイン・ジェム） | SharedPreferences | Key-Value |
| ガチャキャラ一覧 | SharedPreferences | JSON文字列リスト |
| 装備中キャラID | SharedPreferences | 文字列 |
| プレミアム解析の天井カウント | SharedPreferences | 日替わりピックアップSSRを引けなかった連続回数 |
| イベント解析の天井カウント | SharedPreferences | 週替わり限定SSRを引けなかった連続回数 |
| 撃破済み敵リスト | SharedPreferences | 文字列リスト |
| 受取済み実績ID | SharedPreferences | 文字列リスト |
| 受取済みランク報酬ID | SharedPreferences | 文字列リスト |
| ライバルロード進捗 | SharedPreferences | クリア済みステージ数 |
| ライバルロード最短記録 | SharedPreferences | ステージ番号と最短ターンの文字列リスト |
| デイリーミッション状態 | SharedPreferences | 日付、バトル/勝利/ガチャ進捗、受取済みIDリスト |
| 日替わりショップ状態 | SharedPreferences | 日付、購入済み商品IDリスト |
| 週次チャレンジ状態 | SharedPreferences | 週ID、高難度勝利数、受取済みフラグ |
| 期間イベント状態 | SharedPreferences | 週ID、勝利数、受取済み段階報酬ID、最終報酬受取済みフラグ |
| シーズンパス状態 | SharedPreferences | 月ID、シーズンポイント、受取済み報酬IDリスト |
| BOSS最短ターン | SharedPreferences | CPU BOSS勝利時の自己ベスト |
| BOSS撃破報酬日 | SharedPreferences | 最終受取日 |
| キャラ生成シード | SharedPreferences | 整数 |
| バックアップコード | クリップボード/ユーザー保管 | `SPEC-BATTLE-BACKUP:` + Base64url JSON |
| サウンド設定（ミュート） | SharedPreferences | BGM/SE個別の真偽値。バックアップ対象 |

**外部サービス: なし**。完全にオフライン・クライアント完結型。バックエンドサーバー、API、データベースは一切使用しない。

### 5-4. 状態管理の特徴

- **ライブラリ不使用**: Provider/Riverpod/Bloc等を使わず、`StatefulWidget` + `setState` のみ
- **ServiceLocator パターン**: シングルトンコンテナで各サービスを一元管理
- **不変モデル**: すべてのドメインモデルは `copyWith` パターンで更新（ミュータブルな状態変更なし）
- **画面ローカル状態**: 各画面が自分のデータをロード・管理し、戻り時にリロード

## 7. 現時点の制約

### 7-1. 実装上の制約

| 制約 | 詳細 |
|------|------|
| Web版デバイス情報の制限 | ブラウザからCPUコア数・RAM・ストレージを正確に取得できない。フォールバック値を使用 |
| バッテリー機能の無効化 | `battery_plus` がWeb非対応のため、バッテリー連動SPD補正は実質無効（固定値50%） |
| HMAC鍵のハードコード | チェックサム用の秘密鍵 `'SpecBattle_v2_integrity_2026'` がソースコードに埋め込み。Web版ではクライアントサイドで露出 |
| バトル中操作の薄さ | 開始時の支援コマンドと戦術選択はあるが、ターン中のリアルタイム操作は未実装 |

### 7-2. UI/UX上の制約

| 制約 | 詳細 |
|------|------|
| 横画面非対応 | PWA manifest で `portrait-primary` 固定 |
| チュートリアルが薄い | 初回ガイドと遊び方画面はあるが、実戦中の文脈ヘルプは限定的 |
| 空状態UIの残り | 編成画面は空状態CTA対応済み。ほかの一覧・エラー状態は追加確認が必要 |
| オフライン通知なし | ネットワーク状態を検知しない（URL対戦共有時に問題になりうる） |

### 7-3. データ上の制約

| 制約 | 詳細 |
|------|------|
| ローカル保存のみ | ブラウザのlocalStorageに依存。バックアップコード未保存の状態でキャッシュクリアすると全データ喪失 |
| クロスデバイス同期なし | アカウント概念はない。手動バックアップコードによる移行のみ対応 |
| 自動バックアップなし | バックアップ/復元は手動操作。クラウド同期や端末間自動同期は未実装 |
| ガチャキャラのJSON保存 | SharedPreferences に全キャラをJSON文字列として保存。大量保存時のパフォーマンス懸念 |

### 7-4. 運用上の制約

| 制約 | 詳細 |
|------|------|
| 分析基盤なし | ユーザー行動のトラッキング・分析手段がない |
| エラー監視なし | Sentry等のエラー収集サービス未導入 |
| A/Bテスト基盤なし | |
| OGP画像の固定 | 全URLで同一のOGP画像。対戦URLごとの動的OGPは未対応 |

### 7-5. 今後の機能追加でボトルネックになりそうな点

| ボトルネック | 理由 |
|------------|------|
| サーバーレス設計 | リアルタイム対戦、ランキング、アカウント管理にはバックエンド必須 |
| SharedPreferences依存 | データ構造の複雑化に耐えにくい。SQLite等への移行が必要になる可能性 |
| setState のみの状態管理 | 画面間でのデータ共有が手動リロード方式。機能拡張時に状態不整合リスク |
| ガチャキャラ上限なし | **推測**: 所持上限が未定義。数百体以上になると描画・保存のパフォーマンス問題 |

| 弱点 | 改善の方向性 |

| 機能 | 転用案 |

## 10. 要件定義の論点整理

### MVP成立度の評価

**MVPとして成立している部分:**
- ✅ コアループ（キャラ生成 → バトル → 報酬 → 強化）が完結
- ✅ ガチャシステムが機能している（レアリティ・天井・通貨消費）
- ✅ フレンド対戦のURL共有フローが動作する
- ✅ PWA対応でモバイルブラウザからアクセス可能
- ✅ CI/CD でデプロイ自動化済み

**改善済み/まだ粗い部分:**
- ✅ 初回オンボーディングと初回バトル後の次アクション案内がある
- ✅ 敵プレビューの戦術選択とバトル開始時の支援コマンドで最低限の戦略性がある
- ✅ 手動バックアップ/復元でデータ喪失リスクを軽減
- ✅ ジェム消費先としてプレミアム解析・イベント解析・シーズンパス報酬がある
- ⚠️ 課金フロー自体は未実装（決済・アカウント・サーバー保存が必要）
- ⚠️ 分析・モニタリング基盤なし
