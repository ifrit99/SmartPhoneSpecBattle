## 1. プロダクト概要

### このアプリは何をするものか

「SPEC BATTLE」は、**スマートフォンの実スペック（CPU・RAM・ストレージ・バッテリー残量）をキャラクターのステータスに変換し、ターン制オートバトルで戦わせる**ブラウザゲームである。

プレイヤーは自分の端末スペックから自動生成されたキャラクターを使い、CPU敵と戦ったり、URLを共有してフレンドと対戦したりできる。加えて、ガチャシステムにより架空のスマートフォンスペックからキャラクターを獲得・育成する要素もある。

### 想定ユーザー

- スマートフォンに興味のあるカジュアルゲーマー
- 自分の端末スペックを「遊び」に変えたいユーザー
- 友人とURLを共有して気軽に対戦したいユーザー
- **要確認**: ターゲット年齢層・地域は未定義

### 現在提供している主な価値

| 価値 | 説明 |
|------|------|
| 新規性 | 「端末スペック＝キャラ性能」という独自のコンセプト |
| 手軽さ | ブラウザだけで遊べる（インストール不要、PWA対応） |
| ソーシャル性 | URL共有でフレンド対戦が可能 |
| コレクション性 | ガチャで架空端末キャラを収集・図鑑で確認 |

## 2. 主要機能一覧

### 2-1. キャラクター自動生成

| 項目 | 内容 |
|------|------|
| 概要 | 端末のハードウェア情報からキャラクターを決定論的に生成する |
| ユーザーができること | アプリ起動時に自分の端末キャラが自動で作られる |
| 生成ルール | CPU→ATK、RAM→HP、ストレージ空き→DEF、バッテリー残量→SPD |
| 属性 | OS バージョン文字列から6属性（炎・水・地・風・光・闇）の1つを決定 |
| スキル | 属性ごとに固定3つ（攻撃系・防御系・特殊系） |
| 外見 | 頭・体・腕・脚・カラーの組み合わせによるピクセルアート（自動生成） |
| 名前 | 属性ベースの接頭語＋ランダム接尾語（例: フレア・ナイト） |
| 補足 | 同一スペックからは常に同一キャラが生成される（シード値による決定性） |
| 補足（Web） | Web版ではデバイス情報APIの制約あり。取得可能な範囲でフォールバック動作 |

### 2-2. CPU対戦（ソロバトル）

| 項目 | 内容 |
|------|------|
| 概要 | ランダムに生成されたCPU敵とオートバトルで戦う |
| ユーザーができること | ホーム画面の「バトル開始」から対戦。敵プレビュー確認後にバトル開始 |
| 敵の種類 | 16種の架空端末（4難易度×4体）から確率抽選 |
| 難易度分布 | Easy 20%、Normal 50%、Hard 25%、Boss 5% |
| 報酬 | 経験値（勝敗問わず）＋コイン（勝利時に難易度ボーナス加算） |
| バトル方式 | 完全オート、最大50ターン、SPD順で行動、AI行動選択 |
| 補足 | プレイヤーの操作介入はスキップのみ |

### 2-3. フレンド対戦（URL共有）

| 項目 | 内容 |
|------|------|
| 概要 | 自分のキャラデータをURLにエンコードし、相手に共有して非同期対戦する |
| ユーザーができること | ①URLを発行してコピー/共有 ②受け取ったURLを貼り付けて対戦 |
| データ形式 | キャラクターのバイナリデータ→Base64url→URLクエリパラメータ |
| 改ざん防止 | HMAC-SHA256ベースの4バイトチェックサム（v2形式） |
| 後方互換 | v1形式（チェックサムなし）のデータもデコード可能 |
| URL形式 | `https://{host}/?battle={encoded_data}` |
| 補足 | サーバーレス設計。データはURL自体に含まれ、バックエンドは不要 |

### 2-4. ガチャ（エミュレートガチャ）

| 項目 | 内容 |
|------|------|
| 概要 | ゲーム内通貨（コイン）を消費し、架空端末スペックからキャラクターを獲得する |
| ユーザーができること | 1回引き（100コイン）または10連（900コイン、SR以上1体確定） |
| レアリティ | N(60%)、R(25%)、SR(10%)、SSR(5%) |
| ステータス倍率 | N:0.8倍、R:1.0倍、SR:1.2倍、SSR:1.5倍 |
| ガチャプール | 17種の架空端末（N:5、R:5、SR:4、SSR:3） |
| 10連天井 | 最初の9体にSR以上がなければ10体目はSR(67%)/SSR(33%)確定 |
| 補足 | 課金通貨（プレミアムジェム）のUIは存在するが、購入導線は未実装 |

### 2-5. 編成（インベントリ）

| 項目 | 内容 |
|------|------|
| 概要 | ガチャで入手したキャラから1体を「装備」してバトルに使用できる |
| ユーザーができること | キャラ一覧の閲覧、詳細確認、装備/解除 |
| ソート | レアリティ降順 → レベル降順 |
| 補足 | 装備するとホーム画面・バトル・URL共有すべてで装備キャラが使われる |

### 2-6. コレクション（図鑑・戦績）

| 項目 | 内容 |
|------|------|
| 概要 | 過去に遭遇した敵キャラの図鑑と、プレイヤーの対戦成績を閲覧できる |
| ユーザーができること | 敵キャラ図鑑（発見/未発見表示）、戦績（勝率・勝利数等）の確認 |
| 補足 | 未発見キャラは「???」表示で、発見意欲を刺激する設計 |

### 2-7. レベル・経験値システム

| 項目 | 内容 |
|------|------|
| 概要 | バトル結果に応じて経験値を獲得し、レベルアップでステータスが上昇する |
| レベルアップ | 必要経験値 = `100 × (1.0 + (Lv-1) × 0.5)` で漸増 |
| ステータス上昇 | 全ステータスに `1.0 + (Lv-1) × 0.1` の倍率（Lv10で約1.9倍） |
| 経験値獲得 | 勝利: `50 × (1.0 + (敵Lv-1) × 0.2)`、敗北: `20 × 同倍率` |

### 2-8. 通貨システム

| 項目 | 内容 |
|------|------|
| 概要 | バトル報酬としてコインを獲得し、ガチャに使用する |
| コイン獲得量 | 基本: `30 + Lv × 5`、勝利時に難易度ボーナス（+0〜+50） |
| プレミアムジェム | データ構造上は存在するが、獲得・消費の導線は未実装 |

### 2-9. サウンド

| 項目 | 内容 |
|------|------|
| 概要 | BGM（バトル中ループ再生）とSE（攻撃・防御・スキル・勝敗等）を再生 |
| ミュート | BGMとSEを個別にON/OFF可能 |
| Web対応 | AutoPlayポリシー対応済み（初回タップでAudioContext解放） |
| SE再生方式 | Web版はHTML5 Audio APIを直接使用（audioplayers_webのリソース枯渇問題を回避） |

## 4. ユーザーフロー

### 4-1. 初回起動〜初バトル（正常系）

```
1. ブラウザでアクセス
2. タイトル画面表示 → タップ（Web: Audio解放 + BGM開始）
3. ホーム画面表示
   - 端末スペックを読み取り、キャラクターを自動生成
   - キャラカードに名前・属性・レベル・ステータスが表示される
4. 「バトル開始」ボタンをタップ
5. 敵プレビューシート表示
   - 敵キャラの名前・難易度・ステータス・端末情報を確認
6. 「バトル！」をタップ
7. バトル画面に遷移
   - オートでターン進行、ダメージポップアップ・SE演出
   - スキップ可能
8. バトル終了 → 「リザルトへ」をタップ
9. リザルト画面
   - 勝敗、獲得EXP・コイン表示
   - レベルアップ時はアニメーション演出
10. 「ホームへ戻る」→ ホーム画面に帰還（データ更新済み）
```

### 4-2. ガチャ〜キャラ装備

```
1. ホーム画面 → 「Gacha」ボタン
2. ガチャ画面でレート確認
3. 「1回引く」（100コイン）or「10連」（900コイン）をタップ
4. ガチャマシンのシェイク演出
5. 結果ダイアログ表示（レアリティ・キャラ名・ステータス）
6. 「OK」で閉じる → ホーム画面に戻る
7. 「Party」ボタン → 編成画面
8. 獲得キャラをタップ → 詳細シート → 「装備する」
9. ホーム画面に戻ると装備キャラが反映されている
```

### 4-3. フレンド対戦（URL共有側）

```
1. ホーム画面 → 「Friend」ボタン
2. フレンドメニュー → 「URLでシェアする」
3. シェア画面にURL表示
4. 「コピー」or「共有」ボタンでURLを取得
5. SNS/メッセンジャーで友人に送信
```

### 4-4. フレンド対戦（URL受取側）

```
パターンA: URL直接アクセス
1. 受け取ったURL（?battle=xxx付き）をブラウザで開く
2. アプリ起動 → URL解析 → ゲストプレビュー画面に直接遷移
3. 敵キャラ確認 → 「バトル開始」→ バトル → リザルト

パターンB: 手動入力
1. アプリ内でフレンドメニュー → 「URLを入力して対戦」
2. URLまたはコードを貼り付け → 送信
3. ゲストプレビュー → バトル → リザルト
```

### 4-5. 異常系・空状態

| 状況 | 挙動 |
|------|------|
| コイン不足でガチャ | 赤色スナックバーで「コインが足りません」表示、ボタンは無効化されない |
| 不正なURL/コード入力 | 赤色エラーテキスト表示、スナックバーで通知 |
| 改ざんされたURLデータ | `IntegrityException` → エラー表示（チェックサム検証失敗） |
| ガチャキャラ0体で編成画面 | 空のグリッド表示（**要確認**: 空状態の専用UIがあるか） |
| 図鑑で全キャラ未発見 | 全カード「???」表示 |
| バトルが50ターンで決着しない | **要確認**: タイムアウト時の勝敗判定ロジック |

## 5. データ構造・状態管理の概要

### 5-1. 主要データモデル

| モデル | 主要フィールド | 役割 |
|--------|--------------|------|
| Character | name, element, baseStats, currentStats, skills, experience, seed, batteryLevel, 外見インデックス×5 | ゲームの中核。バトル・表示・エンコードすべてに使用 |
| Stats | hp, maxHp, atk, def, spd | 戦闘ステータス（不変値オブジェクト） |
| Skill | name, element, category, multiplier, cooldown, effect | キャラの行動選択肢（属性ごとに3つ固定） |
| StatusEffect | type, duration, value, isPermanent | バフ・デバフ（9種: ATK/DEF/SPD増減、毒、再生、スタン） |
| Experience | level, currentExp, expToNext | レベル進行管理 |
| GachaCharacter | id, deviceName, rarity, obtainedAt, character | ガチャ専用ラッパー（レアリティ情報付き） |
| PlayerCurrency | coins, premiumGems | 通貨管理 |
| DecodedCharacter | character, isGacha, rarity, deviceName | URLデコード結果のラッパー |

### 5-2. データの流れ

```
[端末スペック] → CharacterGenerator → Character
[ガチャプール] → GachaService → GachaCharacter → Character
[Character]   → BattleEngine  → BattleResult（ログ・EXP・コイン）
[Character]   → CharacterCodec → Base64url文字列 → URL
[URL]         → CharacterCodec → DecodedCharacter → Character
```

### 5-3. 永続化

| データ | 保存先 | 方式 |
|--------|--------|------|
| プレイヤーレベル・経験値 | SharedPreferences | Key-Value |
| 戦績（勝利数・バトル数） | SharedPreferences | Key-Value |
| 通貨（コイン・ジェム） | SharedPreferences | Key-Value |
| ガチャキャラ一覧 | SharedPreferences | JSON文字列リスト |
| 装備中キャラID | SharedPreferences | 文字列 |
| 撃破済み敵リスト | SharedPreferences | 文字列リスト |
| キャラ生成シード | SharedPreferences | 整数 |
| サウンド設定（ミュート） | SoundService内部状態 | **推測**: 揮発性（アプリ再起動でリセット） |

**外部サービス: なし**。完全にオフライン・クライアント完結型。バックエンドサーバー、API、データベースは一切使用しない。

### 5-4. 状態管理の特徴

- **ライブラリ不使用**: Provider/Riverpod/Bloc等を使わず、`StatefulWidget` + `setState` のみ
- **ServiceLocator パターン**: シングルトンコンテナで各サービスを一元管理
- **不変モデル**: すべてのドメインモデルは `copyWith` パターンで更新（ミュータブルな状態変更なし）
- **画面ローカル状態**: 各画面が自分のデータをロード・管理し、戻り時にリロード

## 7. 現時点の制約

### 7-1. 実装上の制約

| 制約 | 詳細 |
|------|------|
| Web版デバイス情報の制限 | ブラウザからCPUコア数・RAM・ストレージを正確に取得できない。フォールバック値を使用 |
| バッテリー機能の無効化 | `battery_plus` がWeb非対応のため、バッテリー連動SPD補正は実質無効（固定値50%） |
| HMAC鍵のハードコード | チェックサム用の秘密鍵 `'SpecBattle_v2_integrity_2026'` がソースコードに埋め込み。Web版ではクライアントサイドで露出 |
| プレイヤー操作なしのバトル | 完全オート。戦略的な操作介入ができない |
| ガチャの重複制御なし | **推測**: 同一キャラを複数回獲得可能（合成・売却機能なし） |

### 7-2. UI/UX上の制約

| 制約 | 詳細 |
|------|------|
| 横画面非対応 | PWA manifest で `portrait-primary` 固定 |
| チュートリアルなし | 初回ユーザーへの導線説明がない |
| 空状態UIの不足 | **推測**: ガチャ未実施時の編成画面等に専用の空状態メッセージがない可能性 |
| オフライン通知なし | ネットワーク状態を検知しない（URL対戦共有時に問題になりうる） |

### 7-3. データ上の制約

| 制約 | 詳細 |
|------|------|
| ローカル保存のみ | ブラウザのlocalStorageに依存。キャッシュクリアで全データ喪失 |
| クロスデバイス同期なし | アカウント概念がなく、端末間のデータ移行手段がない |
| バックアップなし | データのエクスポート/インポート機能がない |
| ガチャキャラのJSON保存 | SharedPreferences に全キャラをJSON文字列として保存。大量保存時のパフォーマンス懸念 |

### 7-4. 運用上の制約

| 制約 | 詳細 |
|------|------|
| 分析基盤なし | ユーザー行動のトラッキング・分析手段がない |
| エラー監視なし | Sentry等のエラー収集サービス未導入 |
| A/Bテスト基盤なし | |
| OGP画像の固定 | 全URLで同一のOGP画像。対戦URLごとの動的OGPは未対応 |

### 7-5. 今後の機能追加でボトルネックになりそうな点

| ボトルネック | 理由 |
|------------|------|
| サーバーレス設計 | リアルタイム対戦、ランキング、アカウント管理にはバックエンド必須 |
| SharedPreferences依存 | データ構造の複雑化に耐えにくい。SQLite等への移行が必要になる可能性 |
| setState のみの状態管理 | 画面間でのデータ共有が手動リロード方式。機能拡張時に状態不整合リスク |
| ガチャキャラ上限なし | **推測**: 所持上限が未定義。数百体以上になると描画・保存のパフォーマンス問題 |

## 8. 既知の弱点・改善余地

### 8-1. 継続利用の観点

| 弱点 | 改善の方向性 |
|------|------------|
| バトルが完全オート | プレイヤーの介入要素（コマンド選択、タイミング操作等）の導入 |
| エンドコンテンツの不足 | 高難度ボス、イベントバトル、ランキング等の追加 |
| ソーシャル機能の薄さ | フレンドリスト、対戦履歴の共有、リーダーボード |
| デイリーミッションなし | ログインボーナス、日替わりミッション等の定期的な動機付け |
| 通知なし | プッシュ通知・リマインダー機能がないため再訪動機が弱い |

### 8-2. 収益化の観点

| 弱点 | 改善の方向性 |
|------|------------|
| 課金導線が未実装 | プレミアムジェムの購入フロー構築（決済基盤の選定が必要） |
| 広告枠なし | リワード広告（バトル後のEXP/コインブースト等）の導入 |
| 限定ガチャなし | 期間限定キャラ・イベントガチャによる課金動機の創出 |
| コイン獲得バランス | **要検証**: 無課金でのガチャ回転率が適切かどうか |

### 8-3. 収益化に転用しやすい既存機能

| 機能 | 転用案 |
|------|--------|
| ガチャシステム | プレミアムジェムでの有料ガチャ追加が最も直接的 |
| 通貨システム | コイン/ジェムの二重通貨構造は課金モデルの基盤として成立している |
| レアリティ体系 | 限定SSR等の上位レアリティ追加で訴求力向上 |

### 8-4. 先に改善が必要な基盤

| 基盤 | 理由 |
|------|------|
| データ永続化 | localStorageからの脱却（アカウント＋サーバー保存）が課金の前提 |
| 分析基盤 | ユーザー行動を把握しなければ施策の効果測定ができない |
| エラー監視 | 課金フロー導入前にアプリの安定性を担保する必要がある |

### 8-5. 開発効率の観点

| 弱点 | 改善の方向性 |
|------|------------|
| E2Eテストなし | ブラウザ自動テスト（integration_test等）の導入 |
| Widget テストが少ない | UI変更時のリグレッション検知が弱い |
| 環境変数管理なし | HMAC鍵等のシークレットを環境変数化 |

### 8-6. 保守性の観点

| 弱点 | 改善の方向性 |
|------|------------|
| 画面ファイルの肥大化 | ホーム画面・バトル画面が大きい。Widget分割の余地あり |
| ファイル名とクラス名の乖離 | `qr_menu_screen.dart` → `FriendBattleMenuScreen` 等、リネーム残り |
| 条件分岐コンパイル | Web/Native分岐が散在。プラットフォーム抽象化レイヤーの整理 |

## 10. 要件定義の論点整理

### MVP成立度の評価

**MVPとして成立している部分:**
- ✅ コアループ（キャラ生成 → バトル → 報酬 → 強化）が完結
- ✅ ガチャシステムが機能している（レアリティ・天井・通貨消費）
- ✅ フレンド対戦のURL共有フローが動作する
- ✅ PWA対応でモバイルブラウザからアクセス可能
- ✅ CI/CD でデプロイ自動化済み

**まだ粗い部分:**
- ⚠️ チュートリアル・オンボーディングがない
- ⚠️ バトルに戦略性がない（完全オート）
- ⚠️ データがローカル限定で喪失リスクあり
- ⚠️ 課金フローが未実装（通貨構造だけ存在）
- ⚠️ 分析・モニタリング基盤なし

### 次に議論すべきテーマ

**短期（プロダクト品質向上）:**
- [ ] チュートリアル・オンボーディング導入
- [ ] バトルへのプレイヤー介入要素の検討（コマンド選択、属性相性の可視化等）
- [ ] データ喪失対策（エクスポート/インポート、またはアカウント制）
- [ ] ゲームバランス検証（コイン獲得率、レベルカーブ、ガチャ体験）
- [ ] 空状態・エラー状態のUX改善

**中期（グロース施策）:**
- [ ] ソーシャル機能拡充（フレンドリスト、対戦履歴共有、リーダーボード）
