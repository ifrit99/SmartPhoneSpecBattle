# SPEC BATTLE リリース・収益化評価メモ

作成日: 2026-06-03
対象ブランチ: `fix/review-followups`
目的: Claude Code など別エージェントへ、現行仕様でのリリース判断と将来収益化の見込みを共有する。

---

## 結論

**Web版MVPとしてはリリースしてよい。**

ただし、現仕様のまま即収益化するには弱い。将来的な収益化ポテンシャルはあるが、まずは無料Web MVPとして公開し、継続率・共有率・ガチャ到達率を測る段階。

評価:

- リリース可否: **7.5 / 10**
- 将来の収益化ポテンシャル: **6 / 10**
- 現時点の収益化準備度: **2 / 10**

判断:

- **今すぐ有料課金を入れるべきではない**
- **無料公開してユーザー行動を測るべき**
- **最初の収益化は有料ジェムよりリワード広告が自然**
- **課金ガチャ・有料パスは、アカウント保存と購入復元が入ってから**

---

## リリース可と判断した理由

コアループが成立している。

- 端末/架空端末からキャラクター生成
- CPUバトル
- 報酬獲得
- ガチャ
- 編成/育成
- 図鑑/実績/ローカルランク
- URL共有によるフレンド対戦
- デイリー/週次/月次の継続導線

仕様上も `docs/product_spec.md` でMVP成立要件が整理されている。

- コアループ完結
- ガチャシステムあり
- URL共有フローあり
- PWA対応
- CI/CDデプロイ前提

今回確認した品質ゲート:

```bash
flutter analyze
# No issues found! (ran in 4.5s)

flutter test
# 327件すべて成功
```

補足:

- `flutter analyze` 実行時、iOS側で Swift Package Manager / CocoaPods 移行に関する警告が出る。
- Web MVPとしては致命傷ではない。
- iOS/Androidリリースを再開する場合は別途対応が必要。

---

## 収益化ポテンシャルがある理由

収益化に転用できるゲーム構造はすでにある。

- コイン/ジェムの二重通貨
- 通常ガチャ、プレミアム解析、イベント解析
- SSR、限定SSR、日替わりピックアップ
- 天井
- 重複覚醒
- シーズンパス進行
- 日替わりショップ
- 週替わりイベント
- ロスター収集ボーナス

特に `PlayerCurrency` には、課金通貨に転用しやすい `premiumGems` がある。

```dart
static const int premiumPullCost = 20;
static const int eventLimitedPullCost = 30;
```

現状は無料通貨として配布しているが、構造自体は有料ジェム販売や広告報酬に接続しやすい。

---

## ただし現仕様のまま収益化しにくい理由

### 1. 課金導線が未実装

`docs/product_spec.md` にも明記されている通り、課金導線は未実装。決済、購入履歴、復元、アカウント保存がない。

この状態で有料アイテムを売ると、ブラウザキャッシュ削除で購入物が消えるリスクがある。

### 2. 無料ジェムがかなり潤沢

`test/domain/economy_balance_test.dart` では、以下が回帰基準として固定されている。

- 初回セッションで単発ガチャとイベント解析に届く
- 通常CPU勝利3回で単発ガチャ1回に届く
- 7日継続でプレミアム解析天井とイベント解析天井を両方狙える
- シーズンパス完走報酬は10連2回分以上とイベント解析天井分以上

これは継続率には良いが、有料ジェム販売には不利。

現状のまま課金を追加すると、「払う理由」より「無料で十分」が勝つ可能性が高い。

### 3. 分析基盤がない

現時点では、以下を測れない。

- 初回起動から初回バトルまでの到達率
- 初回ガチャ到達率
- D1 / D7 継続率
- URL共有率
- ガチャ画面での不足表示後の行動
- イベント解析への到達率
- シーズンパス完走率

収益化判断に必要なデータがないため、今の段階で価格や課金導線を決めるのは早い。

### 4. Web版では「端末スペックで戦う」独自性が弱まる

Web版はブラウザ制約により、CPU/RAM/ストレージを正確に取得できずフォールバック値を使う。

このゲーム最大の訴求である「自分のスマホがキャラになる」が、Webでは少し弱い。

無料公開には問題ないが、有料化時は以下のどちらかを選ぶ必要がある。

- Webのまま、架空端末ガチャ/収集ゲームとして割り切る
- ネイティブ版を再開し、実機スペック連動を強める

## 推奨ロードマップ

### Phase 1: 無料Web MVPとして公開

目的は収益ではなく検証。

見るべき指標:

- 初回バトル到達率
- 初回ガチャ到達率
- 2回目バトル率
- URL共有クリック率
- D1 / D7 継続率
- ガチャ不足表示後の離脱率

必要な追加:

- Google Analytics / Firebase Analytics などの軽量分析
- Sentry などのエラー監視
- デプロイ後のOGP/URL対戦/ログイン報酬/バトル報酬確認

### Phase 2: リワード広告

最初の収益化候補。

自然な広告枠:

- バトル後のCoin/EXP 2倍
- 1日1回の追加Gems
- BOSS再挑戦権
- イベント進捗+1
- 日替わりショップの無料更新

理由:

- アカウントなしでも比較的導入しやすい
- ユーザーが報酬目的で能動的に見る形にできる
- 現在の無料ジェム設計と相性が良い

### Phase 3: アカウント/サーバー保存

有料課金の前提。

最低限必要:

- ユーザーID
- 購入履歴
- 所持Gem
- ロスター
- 装備キャラ
- シーズンパス進捗
- 復元フロー

localStorageだけでは有料アイテムを安全に扱えない。

### Phase 4: 有料Gem / 有料シーズンパス

この段階で初めて有料課金を検討する。

候補:

- 有料Gemパック
- 有料シーズンパス報酬列
- 広告非表示
- 限定外装/称号
- 期間限定イベントパック

注意:

- ランダムアイテムを有料販売する場合、Apple / Google Play のルール上、排出率表示が必要。
- ガチャ課金は地域・年齢によって規制リスクがあるため、まずは広告と非ランダム商品が安全。

## Claude Code への引き継ぎメモ

次に実装するなら、課金ではなく以下を優先する。

1. `docs/product_spec.md` の「分析基盤なし」「エラー監視なし」を解消する。
2. Web公開後の計測イベントを設計する。
3. デプロイ後のURL対戦フロー、OGP、ログイン報酬、バトル報酬を手動確認する。
4. リワード広告を入れる場合は、まず広告視聴なしでもゲーム進行できる任意報酬に限定する。
5. 有料Gemや有料ガチャは、アカウント/サーバー保存が入るまで保留する。

避けるべきこと:

- localStorageだけで有料Gemを販売する
- 無料Gem配布量を見直さずに有料Gemを追加する
- 分析なしで価格や課金パックを決める
- Web版のフォールバック端末情報を「実機スペック完全反映」と訴求する
